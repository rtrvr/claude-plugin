# Next Prompt

현재 세션의 작업 맥락을 분석하여, 새 세션에서 바로 사용할 수 있는 핸드오프 프롬프트를 생성합니다.

**입력**: $ARGUMENTS

---

## Step 0: 컨텍스트 수집

핸드오프할 작업의 맥락을 파악합니다.

### $ARGUMENTS가 있는 경우

키워드로 프로젝트 내 관련 파일을 검색합니다:

```
Glob: **/*{키워드}*
Grep: 키워드 관련 패턴
```

검색 결과에서 핵심 파일을 읽어 맥락을 구성합니다.

### $ARGUMENTS가 없는 경우

지금까지의 대화 맥락에서 핸드오프 대상을 추론합니다:

- 현재 작업 중인 과제는 무엇인가?
- 어디까지 완료되었고, 무엇이 남았는가?
- 다음 세션에서 이어받아야 할 핵심은 무엇인가?

추론이 불가능하면 `AskUserQuestion`으로 질문:
- "어떤 작업을 다음 세션으로 넘기고 싶으신가요?"

---

## Step 1: 작업 유형 분류

수집한 맥락을 기반으로 작업 유형을 분류합니다:

| 유형 | 설명 |
|------|------|
| **버그 수정** | 기존 동작의 오류를 수정하는 작업 |
| **기능 구현** | 새로운 기능을 추가하는 작업 |
| **리팩토링** | 동작 변경 없이 코드 구조를 개선하는 작업 |
| **조사/분석** | 원인 파악, 성능 분석, 기술 검토 등 |
| **기타** | 위에 해당하지 않는 작업 |

---

## Step 2: 핸드오프 프롬프트 작성

다음 5대 원칙을 적용하여 프롬프트를 작성합니다.

### 프롬프트 작성 5대 원칙

1. **파일 경로 명시** — 새 세션은 파일을 모른다. 관련 파일의 전체 경로를 반드시 포함한다.
2. **증상과 원인 분리** — "X가 안 된다"(증상)와 "Y 때문이다"(원인)를 구분하여 기술한다.
3. **구조/맥락 간결하게** — 메서드명, 데이터 흐름 등 핵심만 포함한다. 전체 아키텍처 설명은 하지 않는다.
4. **기대 결과 명확하게** — 작업 완료 후 어떤 동작/상태가 되어야 하는지 명시한다.
5. **디버깅 히스토리 제외** — 시도했다 실패한 과정은 빼고, 결론만 전달한다.

### 출력 형식

아래 형식으로 핸드오프 프롬프트를 출력합니다:

```
## 요청 프롬프트

> [작업 유형]: [한 줄 요약]
>
> **관련 파일**
> - `path/to/file` — [역할]
>
> **현재 상태**
> [2-3문장]
>
> **남은 작업**
> - [ ] 태스크 1
> - [ ] 태스크 2
>
> **기대 결과**
> [완료 후 동작 방식]
```

프롬프트 출력 후, 사용자가 복사하여 새 세션에서 바로 사용할 수 있도록 안내합니다.
