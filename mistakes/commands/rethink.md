# Rethink

현재 접근 방식을 백지 상태에서 재검토합니다.

**입력**: $ARGUMENTS

---

## Pre-Step: MISTAKES.md 로드

`.claude/MISTAKES.md` 파일이 존재하면 읽어서 참조합니다.

- 있으면: 과거 실수 패턴을 이번 분석의 체크리스트로 활용
- 없으면: 스킵하고 다음 단계로 진행

---

## Step 0: 컨텍스트 수집

재검토할 주제의 맥락을 파악합니다.

### $ARGUMENTS가 있는 경우

키워드로 프로젝트 내 관련 파일을 검색합니다:

```
Glob: **/*{키워드}*
Grep: 키워드 관련 패턴
```

검색 결과에서 핵심 파일을 읽어 맥락을 구성합니다.

### $ARGUMENTS가 없는 경우

지금까지의 대화 맥락에서 재검토 대상을 추론합니다:

- 현재 논의 중인 문제/과제는 무엇인가?
- 어떤 접근 방식을 취하고 있는가?
- 어디서 막혔거나 불확실한가?

추론이 불가능하면 `AskUserQuestion`으로 질문:
- "어떤 문제나 접근 방식을 재검토하고 싶으신가요?"

---

## Step 1: 백지 상태 분석

> **원칙**: 현재 방향을 잊고, 문제 자체를 처음 보는 것처럼 분석합니다.

### 1-0: 이미 해결된 문제인가?

- 이 문제와 동일하거나 유사한 것이 이미 해결된 적 있는가?
- 기존 해결책, 라이브러리, 도구, 프로세스 중 바로 적용 가능한 것은?
- MISTAKES.md에 관련 패턴이 기록되어 있는가?

### 1-1: 타이밍 분석

- **언제** 이 문제가 발생하는가? (특정 조건, 시점, 순서)
- 항상 발생하는가, 특정 상황에서만 발생하는가?
- 문제의 발생 시점과 인지 시점이 다른가?

### 1-2: 기존 자원 분석

- 이미 있는 것(코드, 도구, 데이터, 인력, 프로세스)만으로 해결 가능한가?
- 현재 방식에서 빠뜨린 기존 자원은 없는가?
- 80% 해결이 가능한 가장 단순한 방법은?

### 1-3: 제약 조건 사고

- 새로운 것을 만들 수 없다면 어떻게 해결하겠는가?
- 시간/예산/인력이 절반이라면?
- 이 제약 조건이 오히려 더 나은 해결책을 만들 수 있는가?

### 1-4: 5 Whys — 근본 원인 파고들기

1. **왜** 이 문제가 발생했는가? → 답변 1
2. **왜** 답변 1이 발생했는가? → 답변 2
3. **왜** 답변 2가 발생했는가? → 답변 3
4. **왜** 답변 3이 발생했는가? → 답변 4
5. **왜** 답변 4가 발생했는가? → **근본 원인**

## Step 2: 현재 방향 vs 대안 비교

최소 2개의 대안을 도출하고 현재 방향과 비교합니다.

| 기준 | 현재 방향 | 대안 A | 대안 B |
|------|----------|--------|--------|
| 핵심 접근 | | | |
| 장점 | | | |
| 단점/리스크 | | | |
| 구현 난이도 | | | |
| 유지보수 부담 | | | |
| 확장성 | | | |

---

## Step 3: 근거 기반 검증 (MANDATORY)

> **추측하지 말고 근거를 찾아라.**
> 모든 주장은 코드, 문서, 데이터, 로그, 공식 레퍼런스 등 확인 가능한 근거로 뒷받침해야 합니다.

### 3-1: 다층 분석

각 계층을 순서대로 검증합니다:

| 계층 | 확인할 것 | 예시 (개발) | 예시 (비개발) |
|------|----------|------------|-------------|
| **현상** | 겉으로 보이는 문제 | 에러 메시지, 버그 | 매출 하락, 이탈률 증가 |
| **원인** | 직접적 원인 | 잘못된 로직, 누락된 처리 | 잘못된 타겟팅, 메시지 불일치 |
| **구조** | 시스템/프로세스 구조 | 아키텍처, 데이터 흐름 | 워크플로우, 조직 구조 |
| **전제** | 깔고 있는 가정 | "이 API는 항상 성공한다" | "고객은 가격에 민감하다" |
| **환경** | 외부 조건/제약 | 프레임워크 제한, 인프라 | 시장 상황, 규제, 예산 |

**각 계층에서 반드시 답할 것:**
- 이 계층에서 확인한 **근거**는 무엇인가?
- 근거 없이 **추측**하고 있는 부분은 무엇인가?
- 추측을 근거로 바꾸려면 무엇을 확인해야 하는가?

### 3-2: 영향 범위 / 의존 관계 확인

- 이 문제 또는 해결책이 영향을 미치는 범위는?
- 다른 구성 요소, 팀, 프로세스에 미치는 파급 효과는?
- 숨겨진 의존 관계가 있는가?

### 3-3: 인과 관계 검증

- A → B라고 생각하는 인과 관계가 실제로 성립하는가?
- 상관관계를 인과관계로 착각하고 있지 않은가?
- 역방향(B → A)이나 제3의 원인(C → A, C → B)은 없는가?

### 3-4: 검증 체크리스트

- [ ] 모든 주장에 근거가 있는가?
- [ ] "~일 것이다", "아마", "보통은" 같은 추측 표현이 남아있지 않은가?
- [ ] 현상과 원인을 혼동하고 있지 않은가?
- [ ] 확인하지 않은 전제가 있지 않은가?
- [ ] 영향 범위를 과소/과대평가하고 있지 않은가?

---

## Step 4: 대안 없을 때 자가 점검

Step 2에서 의미 있는 대안이 나오지 않았다면 다음을 점검합니다:

1. **프레이밍 오류**: 문제 자체를 잘못 정의하고 있지 않은가?
2. **앵커링**: 처음 떠올린 방법에 고착되어 있지 않은가?
3. **전문가 편향**: 자기 전문 분야의 도구로만 해결하려 하고 있지 않은가?
4. **매몰 비용**: 이미 투자한 시간/노력 때문에 방향 전환을 꺼리고 있지 않은가?

---

## Step 5: 결론 + 사용자 확인

분석 결과를 다음 형식으로 정리합니다:

```
## 🔄 Rethink 결과

### 현재 접근 방식
[현재 방향 요약]

### 핵심 발견
- [근거 기반 검증에서 발견한 핵심 사항 1]
- [핵심 사항 2]
- ...

### 권장 방향
[현재 유지 / 대안 A / 대안 B] — [이유]

### 주의할 점
- [리스크 또는 추가 검증 필요 사항]
```

`AskUserQuestion`으로 사용자에게 확인:
- "이 분석 결과를 바탕으로 [권장 방향]으로 진행할까요?"
- 사용자가 다른 방향을 원하면 해당 방향에 대해 Step 3 검증을 재실행

---

## Step 6: 실수 패턴 기록

이번 rethink에서 실수 패턴이 발견되었다면 MISTAKES.md에 기록합니다.

**카테고리** (`add-mistake`와 동일):
- `assumption`: 잘못된 가정 또는 검증 없는 추측
- `scope`: 영향 범위 과소/과대 평가
- `timing`: 순서, 타이밍, 시점 관련 오류
- `lifecycle`: 생명주기, 상태 관리, 프로세스 흐름 오류

**형식:**
```
- **제목** [domain-tag] -- 설명. (출처: rethink)
```

실수 패턴이 없으면 이 단계를 스킵합니다.
